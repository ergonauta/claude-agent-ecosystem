# Workflow — Detailed (Phase by Phase)

This document is the operational reference. It specifies exactly what Claude does,
what the human does, what files are read/written, and what constitutes "done"
at each gate.

---

## Before You Start: Prerequisites

Every session begins by loading three files:

```
1. implementation-plan.md      ← The contract (read-only after approval)
2. implementation-status.md    ← Current state (updated after each gate)
3. context/repo-context.md     ← Codebase conventions (generated by /repo-cartographer)
```

If `repo-context.md` does not exist, run `/repo-cartographer` first.

---

## Phase 0 — Planning Session

### What Claude does
- Enters planning mode (no code, no files written yet)
- Asks the human to describe the feature
- Actively challenges the description:
  - "What happens when X is null?"
  - "Is this behaviour expected for unauthenticated users?"
  - "This acceptance criterion conflicts with the existing behaviour in Y"
  - "You haven't specified the error state for Z"
- Helps translate high-level descriptions into precise acceptance criteria
- Proposes phase breakdown (each phase = one atomic layer of work)
- Identifies risks and flags them explicitly

### What the human does
- Drives the feature description
- Answers challenges
- Refines until the spec is unambiguous
- Approves the final plan structure

### Files written
- `implementation-plan.md` — written collaboratively, approved by human, then frozen
- `implementation-status.md` — created with Phase 0 marked complete

### Gate: Plan Approval
Human explicitly approves the plan.
After this point, `implementation-plan.md` is **read-only**.
Any scope change requires a new planning session and a plan amendment.

## Phase 1 — Plan Review

### What Claude does
Reads `implementation-plan.md` in full.
Reviews through two lenses simultaneously:

**Tech Lead lens**
- Are phases correctly scoped? (Not too large to commit atomically, not so small they're trivial)
- Are there missing technical considerations? (Error handling, auth, performance, existing patterns)
- Does the approach align with `repo-context.md`? (Naming, structure, libraries in use)
- Are there architectural risks in the proposed approach?
- Are there dependencies between phases that could cause integration problems?

**PM lens**
- Does each acceptance criterion represent observable user/system behaviour?
- Are there missing scenarios? (Happy path only? What about degraded states?)
- Is scope creep embedded in the requirements? (Anything that wasn't in the original ask)
- Are success criteria measurable?

**Output format**
```markdown
## Plan Review — [Feature Name]

### Tech Lead Findings
- [BLOCKING] <finding> — <recommendation>
- [NOTE] <observation>

### PM Findings
- [BLOCKING] <finding> — <recommendation>
- [NOTE] <observation>

### Approved phases
- Phase 1: <name> — <scope summary>
- Phase 2: <name> — <scope summary>
...
```

### What the human does
- Reviews each finding
- Resolves BLOCKING items (amend plan or explicitly accept risk)
- Notes any NOTEs that should become follow-up tickets

### Gate: Plan Review Approval
Human resolves or explicitly accepts all BLOCKING items.
`implementation-status.md` updated: Phase 1 complete.

## Phase 2 — RED (Per Feature Phase)

> This phase runs once per feature phase defined in the plan.
> If the plan has 3 implementation phases, RED runs 3 times.

### What Claude does
- Reads the current phase definition from `implementation-plan.md`
- Translates each acceptance criterion into unit tests
- Tests are written to **fail** — they describe intent, not implementation
- Test descriptions are human-readable plain English
- Tests cover: happy path, edge cases, failure modes, boundary conditions
- Does NOT write any implementation code

### Test writing rules
- One test per acceptance criterion (minimum)
- Test names describe behaviour: `it('returns 401 when token is expired')`
- No implementation details in tests — test the interface, not the internals
- Tests use the existing test framework and patterns from `repo-context.md`

### What the human does at the gate
**Run the tests. Confirm they all fail.**

Then review:
- [ ] Every acceptance criterion for this phase has at least one test
- [ ] Edge cases from the plan review findings are covered
- [ ] Failure modes are tested, not just success paths
- [ ] Test names clearly describe expected behaviour
- [ ] Tests fail for the *right reason* (not a missing import or syntax error)
- [ ] No test is asserting implementation internals

### Gate: RED Approval
Human confirms tests fail correctly and approves.
`implementation-status.md` updated: RED for Phase N complete.

**Context is cleared after this gate.**
Next session loads: plan + status + repo-context only.

---

## Phase 3 — GREEN (Per Feature Phase)

### What Claude does
- Reads `implementation-plan.md`, `implementation-status.md`, `repo-context.md`
- Implements the minimum code to make the failing tests pass
- Follows conventions from `repo-context.md` strictly
- Does NOT add untested functionality
- Does NOT refactor outside the scope of the current phase
- After each sub-step within the phase: commits atomically before continuing

### Atomic commit rule during GREEN
Each logical unit of work within a phase is committed before the next begins.
This prevents the LLM drift failure mode: working code is locked before proceeding.

### What the human does at the gate
**Run the full test suite.**

Then review:
- [ ] All RED tests for this phase now pass
- [ ] No previously passing tests are now failing
- [ ] Implementation matches the plan (check against acceptance criteria)
- [ ] Nothing extra was implemented beyond what tests cover
- [ ] Code follows patterns from `repo-context.md`

### Gate: GREEN Approval
Human confirms all tests pass and approves implementation.
`implementation-status.md` updated: GREEN for Phase N complete.

## Phase 4 — Self Review

### What Claude does (`/self-review`)
- Reads the staged diff (or branch diff against main)
- Reviews as an adversarial senior developer — explicitly looking for reasons to reject
- Produces a structured feedback list

**Review dimensions:**
- Correctness — does it do what it claims?
- Edge cases — what's unhandled?
- Security — any obvious vulnerabilities introduced?
- Performance — any obvious regressions?
- Readability — would another developer understand this in 6 months?
- Consistency — does it match the patterns already in the codebase?
- Test quality — are the tests actually meaningful?

**Output format:**
```markdown
## Self Review — Phase N: [Phase Name]

### Blocking
- [ ] <issue> — <location> — <recommendation>

### Non-Blocking
- [ ] <observation> — <recommendation>

### Approved (no action needed)
- <item>
```

### What the human does at the gate
- Reviews each blocking item
- Decides: resolve now, log as issue, or explicitly accept
- Notes non-blocking items that warrant follow-up

### Gate: Self Review Approval
All blocking items resolved or explicitly accepted.
`implementation-status.md` updated: Self Review for Phase N complete.

---

## Phase 5 — Commit

### What Claude does (`/atomic-commits`)
- Reviews all staged changes
- Groups changes into logical atomic units
- Writes commit messages following Conventional Commits spec
- Each message references the plan phase

**Commit message format:**
```
<type>(<scope>): <description>

[optional body]

Refs: implementation-plan.md Phase N
```

**Proposes commits to human before executing.**

### Gate: Commit Approval
Human reviews proposed commit messages and approves.
Claude executes commits.

---

## Phase 6 — Pull Request

### What Claude does (`/gh-create-pr`)
- Reads `implementation-plan.md` and `implementation-status.md`
- Drafts PR from `templates/pr-template.md`
- Populates: summary, changes per phase, test coverage notes, ADR decisions made

**PR is drafted — not opened — until human approves.**

### Gate: PR Approval
Human reviews PR draft, edits if needed, approves opening.

---

## Phase ADR Note (After Every Gate)

After each gate approval, Claude appends a brief ADR-style note to `implementation-status.md`:

```markdown
## Phase N ADR Note — [date]
**Decision:** <what was decided or implemented>
**Rationale:** <why this approach, not alternatives>
**Trade-offs:** <what was accepted or deferred>
```

This ensures the *why* is never lost across context clears.

---

## Abort Protocol

If at any gate the human finds the work significantly misaligned:

1. Do NOT continue to the next phase
2. Mark the phase as FAILED in `implementation-status.md`
3. Note what was wrong and why
4. Return to the last approved state (git reset if needed)
5. Re-open planning for that phase only

This prevents compounding errors and wasted review cycles.
