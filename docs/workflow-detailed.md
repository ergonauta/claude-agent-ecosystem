# Workflow — Detailed (Phase by Phase)

This document is the operational reference. It specifies exactly what Claude does,
what the human does, what files are read/written, and what constitutes "done"
at each gate. Read `workflow.md` first for the high-level shape.

---

## Before You Start: Prerequisites

Every session begins by loading:

```
1. feature-spec.md              ← Human-written feature description (before planning)
2. implementation-plan.md       ← The contract (read-only after G1)
3. implementation-status.md     ← Current state (updated after each gate)
4. context/repo-context.md      ← Codebase conventions (generated by /repo-cartographer)
```

`feature-spec.md` must exist before planning begins.
`implementation-plan.md` and `implementation-status.md` are created during planning.
If `repo-context.md` does not exist, run `/repo-cartographer` first.

---

## Stage 1 — Feature Spec

### What the human does
- Writes `feature-spec.md` from `templates/template-feature-spec.md`
- Describes the feature in their own words — no formal format required
- Can reference any source: Jira tickets, ADRs, Slack threads, conversations
- Must consolidate everything into a single document before the planning session

### What Claude does
- Nothing yet. Does not read or challenge the spec at this stage.

### Done when
Human considers the spec complete enough to hand to Claude for planning.

---

## Stage 2 — Planning

**No code. No file changes until `implementation-plan.md` is approved.**

### Step 1 — Stack detection

Claude reads `repo-context.md` and identifies the tech stack.
Notes which specialist agents apply to this feature:

| Agent | When to invoke |
|---|---|
| `/react-agent` | React or Next.js project |
| `/fastapi-agent` | FastAPI project |
| `/tech-lead-agent` | Always |
| `/architect-agent` | Always |

If no specialist agent exists for the detected stack, Claude proceeds without one
and flags the gap in the plan under Risks.

### Step 2 — Spec challenge

Each relevant agent reviews `feature-spec.md` independently from their lens:

**Stack specialist(s):**
- Are the described behaviours achievable with this stack?
- Are there missing constraints specific to the framework?
- Are acceptance criteria testable with this stack's standard tools?

**`/tech-lead-agent`:**
- Are shared concerns (auth, config, DB schema) implicit in the spec but unresolved?
- Does the spec account for backwards compatibility if deployed incrementally?
- Are observability requirements missing?

**`/architect-agent`:**
- Does the spec respect existing service and module boundaries?
- Does the described data flow introduce coupling that's hard to undo?
- Does this feature require an ADR of its own before implementation begins?

Claude consolidates all agent challenges into a single structured list and presents
it to the human. **Does not proceed to Step 3 until every challenge is resolved
or explicitly accepted.**

### Step 3 — Phase breakdown and PR grouping

Claude proposes phases and, in the same step, identifies PR boundaries by concern.

**Phase breakdown rules:**
- Each phase is one coherent, atomic layer of work
- Ordered to minimise integration risk
- No phase larger than can be reviewed in a single sitting

**PR grouping — ask per layer:**
*What can be merged and used independently, before the rest of the feature exists?*

| Concern layer | What belongs here |
|---|---|
| Generic / shared | Reusable components, utilities, data structures — no feature-specific knowledge |
| Domain / data | Business logic, parsing, transformations — depends on generic layer |
| Integration | Feature-specific assembly — wiring components with domain logic, routing |

If more than one PR is identified, Claude invokes `/stacked-branches` to formalise
the branching plan. This is embedded into `implementation-plan.md` before G1.

### Files written
- `implementation-plan.md` — from `templates/template-implementation-plan.md`
- `implementation-status.md` — from `templates/template-implementation-status.md`

### Gate G1 — Plan Approval

```
GATE G1: Plan Approval
Status: AWAITING APPROVAL

Stack detected: [stack]
Specialist agents consulted: [list]
Branching strategy: [single PR | stacked — N PRs]

Checklist:
- [ ] All acceptance criteria are specific and testable
- [ ] Out-of-scope items are explicitly listed
- [ ] Failure modes are documented
- [ ] Phase breakdown is appropriately granular
- [ ] Risks and open questions are captured
- [ ] PR strategy is appropriate for the scope

To approve: respond "APPROVED"
To request changes: respond with what needs to change
```

After approval: `implementation-plan.md` is **read-only** — except via `/plan-revision`.

## Per-Phase Loop

The following stages (RED → GREEN → SELF-REVIEW → COMMIT) run once per phase.
If the plan has 3 phases, these four stages run 3 times before PR.

Context is cleared between stages. Each new session reloads:
`implementation-plan.md` + `implementation-status.md` + `repo-context.md`

---

## Stage 3 — RED (per phase)

### What Claude does
- Reads the current phase definition from `implementation-plan.md`
- Translates each acceptance criterion into tests written to **fail**
- Test descriptions are human-readable plain English
- Covers: happy path, edge cases, failure modes, boundary conditions
- Does **not** write any implementation code

### Test writing rules
- One test minimum per acceptance criterion
- Test names describe behaviour: `it('returns 401 when token is expired')`
- No implementation details in tests — test the interface, not internals
- Tests use the existing framework and patterns from `repo-context.md`

### What the human does at the gate
**Run the tests. Confirm they all fail.**

### Gate G3 — RED Approval

```
GATE G3: RED Approval — Phase [N]
Status: AWAITING APPROVAL

Checklist:
- [ ] Every acceptance criterion for this phase has at least one test
- [ ] Edge cases and failure modes are covered
- [ ] Test descriptions are human-readable and match intent
- [ ] No test checks implementation details — behaviour only
- [ ] Tests run and fail for the right reason (not syntax errors or missing imports)

To approve: respond "APPROVED"
To reject: respond "REJECTED — [reason]"
```

After approval: `implementation-status.md` updated — RED Phase N ✅.
Context cleared. Next session reloads plan + status + repo-context.

---

## Stage 4 — GREEN (per phase)

### What Claude does
- Implements the minimum code to make the failing tests pass — no more
- Follows conventions from `repo-context.md` strictly
- Does **not** add untested functionality
- Does **not** refactor outside current phase scope
- After each logical sub-step within the phase: proposes a commit via `/atomic-commits`
  before continuing (see Commit format below)

### Atomic commit rule during GREEN
Each logical unit of work is committed before the next begins.
Prevents LLM drift: working code is locked before proceeding.

### What the human does at the gate
**Run the full test suite.**

### Gate G4 — GREEN Approval

```
GATE G4: GREEN Approval — Phase [N]
Status: AWAITING APPROVAL

Checklist:
- [ ] All RED tests for this phase now pass
- [ ] No previously passing tests are now failing
- [ ] Implementation matches the plan (not a creative interpretation)
- [ ] Nothing extra implemented beyond what tests cover
- [ ] Code follows patterns from repo-context.md

To approve: respond "APPROVED"
To reject: respond "REJECTED — [reason]"
```

After approval: `implementation-status.md` updated — GREEN Phase N ✅.

---

## Stage 5 — Self Review (per phase)

### What Claude does (`/self-review`)

Assembles a five-persona review panel. Each persona reviews the staged diff
independently. Claude consolidates all findings and writes a single summary file.

**Review panel:**

| Persona | Lens |
|---|---|
| Senior [Stack] Engineer | Correctness, idiomatic patterns, performance, framework-specific pitfalls |
| Tech Lead | Scope adherence to plan, cross-cutting concerns, delivery risk |
| Architect | Boundary violations, coupling introduced, structural regressions |
| QA Engineer | Test coverage against ACs, edge cases, whether tests give real confidence |
| Security Engineer | OWASP top 10, auth, injection, data exposure, unvalidated inputs |

**Output:** `self-review-phase-N.md` written from `templates/template-self-review.md`

All findings use **Conventional Comments** format:
`issue (blocking)`, `suggestion (non-blocking)`, `question`, `nitpick`, `praise`, etc.
Source persona noted inline per finding.

Summary structure:
- **Must Resolve** — `issue`/`todo`/`chore` with `(blocking)` decoration
- **Should Consider** — `suggestion`, `question`, non-blocking items
- **Notes** — `praise`, `thought`, `note` — max 3 items

### What the human does at the gate
- Reads `self-review-phase-N.md`
- For each Must Resolve item: fix it, or accept with explicit reason

### Gate G5 — Self Review Approval

```
GATE G5: Self Review Approval — Phase [N]
Status: AWAITING APPROVAL

Summary: self-review-phase-N.md
Reviewers: Senior [Stack] Engineer · Tech Lead · Architect · QA Engineer · Security Engineer

Must Resolve: [N items]
Should Consider: [N items]

For each Must Resolve item:
- Fix it, OR
- Accept explicitly: "accepting [item] — [reason]"

Questions must be answered before approval.

To approve: respond "APPROVED"
To request changes: respond with which items need resolution
```

After approval: `implementation-status.md` updated — Self Review Phase N ✅.
Append ADR note (see ADR Note Protocol below).

## Stage 6 — Commit (per phase)

### What Claude does (`/atomic-commits`)

Proposes commits for all sub-steps in the phase following **Conventional Commits v1.0.0**.

**Format:**
```
<type>[optional scope][optional !]: <description>

[optional body]

[optional footer(s)]
Refs: implementation-plan.md Phase N
```

**Types:** `feat`, `fix`, `test`, `refactor`, `docs`, `chore`, `perf`, `style`, `ci`, `build`

**Breaking changes:** append `!` to type, or add `BREAKING CHANGE:` footer (MUST be uppercase)

**Rules:**
- Imperative mood: "add expiry check" not "added expiry check"
- Description under 72 characters, no trailing period
- Body explains *why*, not *what* — the diff shows what
- `Refs:` footer always present
- Claude **never commits autonomously** — every commit proposed before executing

### Gate G6 — Commit Approval

```
GATE G6: Commit Approval — Phase [N]
Status: AWAITING APPROVAL

Proposed commits:
  1. test(auth): add failing tests for JWT expiry validation
     Refs: implementation-plan.md Phase 1

  2. feat(auth): implement JWT expiry check in middleware
     Refs: implementation-plan.md Phase 1

To approve: respond "APPROVED"
To revise: specify which commit needs changes
```

After approval: commits executed. Move to next phase's RED, or to PR if all phases done.

## Plan Revision — Escape Hatch

**Trigger:** A phase (during RED or GREEN) reveals a spec problem — not an implementation
problem. The plan is wrong, not the code.

**Do not** invoke for implementation issues. Fix the implementation.
**Do** invoke when an acceptance criterion is wrong, a phase assumption is false,
or a requirement has changed.

### What Claude does (`/plan-revision`)
1. States precisely: which phase, which AC, what the plan says, what is wrong
2. Traces root cause: is the error in `implementation-plan.md` or `feature-spec.md`?
3. Proposes the minimum targeted changes to resolve the gap
4. Presents Gate G1-R before overwriting anything

### Gate G1-R — Plan Revision Approval

```
GATE G1-R: Plan Revision Approval
Status: AWAITING APPROVAL

Gap identified in: Phase [N] — [description]
Root cause: [implementation-plan.md | feature-spec.md]
Phases affected: [list]

Proposed changes:
[before/after or diff]

To approve: respond "APPROVED"
To reject: respond "REJECTED — [reason]"
To update spec first: respond "UPDATE SPEC — [what needs clarifying]"
```

After approval: `implementation-plan.md` overwritten. `implementation-status.md`
updated with G1-R record and ADR note. Resume from RED for the affected phase.

---

## Stage 7 — PR

**Triggered when:** all phases committed and approved.

### What Claude does (`/gh-create-pr`)
- Reads `implementation-plan.md` and `implementation-status.md`
- Drafts PR from `templates/template-pr.md`
- Populates: summary, AC coverage table, commit list, testing notes, ADR decisions
- For stacked PRs: opens one PR per stack level in dependency order

**PR is drafted — not opened — until human approves.**

### Gate G7 — PR Approval

```
GATE G7: PR Approval
Status: AWAITING APPROVAL

Checklist:
- [ ] PR description complete — no empty sections
- [ ] AC coverage table accurate
- [ ] Commit list correct
- [ ] No unresolved blocking items from self-reviews

To approve: respond "APPROVED"
To adjust: respond with what to change
```

## ADR Note Protocol

After every gate approval, Claude appends a brief note to `implementation-status.md`:

```markdown
### Phase N ADR Note — [date] — [Gate]
**Decision:** [what was decided or implemented]
**Rationale:** [why this approach]
**Trade-offs:** [what was accepted or deferred]
```

This ensures the *why* survives context clears and is available for retrospectives,
PR descriptions, and future work on the same area.

---

## Abort Protocol

If at any gate the work is significantly misaligned:

1. Do **not** continue to the next phase
2. Determine the root cause:
   - **Spec problem** → invoke `/plan-revision` (preferred path)
   - **Implementation problem** → fix the implementation, re-run gate
   - **Fundamental misalignment** → mark phase FAILED in `implementation-status.md`,
     note what was wrong and why, git reset to last approved state, re-open planning
     for that phase only
3. Never compound errors by proceeding past a failed gate
